4) LiveTick tight loop hypothesis

If logs show “Next tick … (in -Xs)” repeating, that’s a real bug: when scheduled time is in the past, the scheduler should immediately compute the next future tick and schedule once—not spin/log repeatedly.

What’s inconsistent / likely wrong ⚠️
A) They contradict themselves on ingestion write safety

Earlier, Replit claimed ingestion is already safe: upsert in a single transaction, stale cleanup at end, rollback preserves old data, etc.

Now they claim:

“delete-then-insert… if crashes after deletion you get an empty table.”

Those two can’t both be true. Either:

the write is transactional/upsert-safe (then a crash shouldn’t leave it empty), or

there is a delete-first path (or a non-transactional cleanup) that can empty the table.

Action: ask Replit to point to the exact SQL / code path that deletes rows in trending_people, and show whether it’s inside the same transaction as the upsert/write.

B) “Snapshot fallback is working for leaderboard (serving 100)”

…but you said both Live and Full show N/A, and movers are empty. That suggests:

either fallback is not truly wired into the UI paths you’re using now, or

only one endpoint has fallback and the rest don’t (likely), or

the client expects freshness fields only from trending_people and doesn’t read fallback meta.

So the fix is not just “fallback exists”; it’s make fallback consistent across endpoints + provide freshness timestamps from snapshots/ingestion_runs when trending_people is empty.

C) GDELT being slow shouldn’t make trending_people empty

GDELT delays can cause no new data, but it shouldn’t wipe your existing last-good dataset. Empty table implies either:

a cleanup process ran, or

a migration reset, or

ingestion write logic wiped or truncated, or

another component wrote trending_people incorrectly.
