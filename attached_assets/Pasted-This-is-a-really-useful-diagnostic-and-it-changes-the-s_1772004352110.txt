What that “40” floor likely means

Replit is basically describing a classic missing-data sentinel (a default/fallback value) being injected into scoring:

When search data is stale/missing → set search score to a baseline (40)

When refresh happens → people jump up/down depending on new values

Since search is a heavy velocity weight, that causes broad score shifts

So yes, it’s explainable — but it’s also a design bug from a product/trust perspective.

What I’d recommend as “the way forward”

You don’t need to redesign the whole engine. You just need to stop hard snapping.

Fix option A (best): “Carry forward + decay” instead of snapping to 40

When fresh search data is unavailable:

keep last known search value (carry forward)

apply a gentle decay over time if it stays stale (e.g., drift toward baseline slowly, not instantly)

This prevents the entire leaderboard from lurching on refresh hours.

Fix option B (simple): Search EMA hold when missing/stale

You already have the concept of EMA holds for artifacts.
Do the same for search:

if search freshness is bad → hold previous EMA’d search component

do not inject a raw default into the scoring path

Fix option C (quick UI mitigation): “Data refresh hour” dampener

If you detect an hour where an unusually high % of people changed search values (like the hour Replit found):

temporarily reduce search weight for that tick (like the news governor concept)

or mark it as a “refresh tick” and smooth harder

That’s not as clean as A/B, but it can be a fast patch.
