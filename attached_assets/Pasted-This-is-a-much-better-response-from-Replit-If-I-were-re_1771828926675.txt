The 3 things I’d verify right now
1) Are the env vars actually being read everywhere?

Make sure:

defaults are used when env missing

values are parsed as numbers (not strings), and invalid values don’t silently become NaN

the same values are used by both chart and rank calculations (no “half-old / half-new” behavior)

Quick test: set EMA_BASE_ALPHA=0.12 and confirm you truly revert to prior behavior.

2) The “rank churn” query must compare the same cohort

This is the classic trap.

If your “top 100 tracked people” is stable, fine. But if the set changes (new person added, person removed, missing snapshots, gaps), churn can look worse/better than it really is.

Rank churn should be computed on:

intersection of people present in both snapshots

the same leaderboard slice (e.g., top 100 tracked)

same timestamp alignment (hour-to-hour)

If they didn’t do intersection logic, you’ll get fake churn during ingestion gaps or roster changes.

3) Make sure you’re not measuring “rank churn from noise”

Add one more stat (tiny but powerful):

Churn excluding tiny moves (e.g., only count rank changes where score moved > 0.5% or rank moved ≥ 3 spots)

Because a leaderboard can “shuffle” due to micro differences even when nothing meaningful is happening. That’s the stuff that creates “whiplash” perception.

One copy-level / UX implication

If Trump (and other top people) starts moving more visibly, you may want to ensure users have:

the “Live / Full” timestamps (you already do)

and your “Hot Movers” logic isn’t overly sensitive (so it doesn’t become “Hot Movers = everyone”)

The monitoring “big movers >5% hourly” helps here — watch that number like a hawk.
