## 1) Are Replit’s 3 fixes the right approach?

### ✅ A) Retry + timeout + optional SSL relax for GDELT

Good. Transient TLS/cert/timeouts are common with GDELT endpoints or network paths.

* **Retries + backoff + timeout** prevents a slow request from ruining an ingestion run.
* **Env-flag SSL bypass** is the correct way to keep it from being “always on.”

**One caution:** SSL bypass should be an “emergency lever” only. It’s better to use it temporarily while you troubleshoot root cause.

### ✅ B) Graceful degradation (carry forward last known values)

This is the correct antidote to “everyone goes to zero.”
If a source fails, you don’t want to write zeros and accidentally rewrite reality.

The key part is what they added:

> treat “failure OR suspiciously low values (90%+ drop)” as degraded and fill-forward.

That’s exactly the right mental model: **0 can be a data bug, not reality.**

### ✅ C) Recovery mode (temporarily loosen caps when data returns)

Also correct — without it you get “source comes back but the system can’t recover fast enough” (because your stabilizers slow everything down).

---

## 2) The two big risks in this design (and how to handle them)

### Risk #1: Fill-forward can mask real drops

If someone genuinely stops trending, but the source is stale and you keep carrying forward, you can artificially “prop them up” for a while.

**Mitigation (simple + important):**
Add **staleness decay**:

* Hour 0–2 stale: 100% fill-forward
* Hour 3–6 stale: gradually reduce that source’s contribution (e.g., 80% → 50%)
* > 6–12h stale: either down-weight heavily or exclude that source for scoring, and renormalize weights

This way you don’t “freeze the world,” but you also don’t tank rankings instantly.

### Risk #2: Your “90% drop = suspicious” rule can false-trigger

A 90% drop might be real for some people or some sources (especially search/news).
If that rule triggers too often, you’ll overuse fallback, and your engine becomes “sticky.”

**Mitigation: make the anomaly rule smarter:**
Instead of only “90% drop”, require **at least 2 conditions**:

* `drop > 90%` **AND**
* source-wide: “many celebs collapsed” (e.g., >70% of the leaderboard saw near-zero) **OR**
* request-level: the API returned an error/timeout/empty payload pattern **OR**
* absolute floor: values below a minimum plausible threshold for that celeb (based on their baseline p50/p75)

This will correctly classify the “global zeros” event as a data outage while still allowing genuine individual drops.

## 3) The 3 improvements that will make this bulletproof

### A) Add a **Source Health** state machine (per source)

Right now you have: fresh vs fallback and “recovery mode.”
Make it explicit:

* **HEALTHY**
* **DEGRADED** (errors/spiky zeros)
* **OUTAGE** (too many consecutive failures or global-zero detection)
* **RECOVERY** (first N runs after outage)

Log it like:
`[SOURCE:NEWS] state=OUTAGE reason=global_zero`
This makes debugging instant.

### B) Show a UI banner when degraded data is in play

This matters for trust.

If you are fill-forwarding news/search, show:

* “⚠️ Some data sources are temporarily delayed. Rankings may be less responsive.”

And in your new “Why did this move?” panel, show:

* which sources were **fresh vs fallback**
* how stale they are (minutes since last success)

### C) Don’t just “carry forward raw values” — carry forward **normalized contribution**

Because you now normalize by 7-day percentiles, it’s sometimes cleaner to fill-forward the **source contribution** (or at least store the normalized percentile used).

Why?

* If population stats drift while you’re stale, refitting raw → percentile can create unexpected movement.

Not mandatory today, but it’s the next level of stability.

## 4) What likely caused Serper to go to 0 as well?

Replit focused on GDELT cert/timeouts (credible), but both being 0 suggests a broader issue too:

Common causes:

* Serper quota/rate limiting leading to empty/blocked responses
* request param bug (changed query format)
* caching returning empty payload cached as “valid”
* a shared upstream network issue

**Recommendation:** treat “global zeros” as an outage condition regardless of whether the HTTP call “succeeded.”
