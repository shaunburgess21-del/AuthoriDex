The 3 things I’d ask Replit to verify / adjust 🔧
A) “Top-25” should be based on current rank, not “displayOrder”

Replit says: “sort by displayOrder.” That may be fine if displayOrder == rank (or is always kept consistent). But in most systems:

displayOrder = UI ordering, maybe curated

rank = dynamic, based on fame_index

If you want top-25 ranked, compute top-25 using current leaderboard rank (from trending_people.rank after the last run), or at least from tracked_people.rank if that’s your canonical rank source.

Tell Replit:

B) Confirm newsSource tagging is truly present in recent snapshots

Replit says it’s already there. Your earlier health check said it was unset. That could mean:

old snapshots from before the fix,

or a code path that writes diagnostics differently (fallback path vs normal path),

or the UI reads a different diagnostics field.

Simple verification request:

Please run a query on the latest completed run’s snapshots and show the distribution of diagnostics.newsSource values (mediastack/gdelt/serper_news/cache/undefined). We should see near-100% non-null now.

C) Make sure persistence is done “after commit”

If they write system:lastRunMeta inside the same transaction before the run is fully finalized, a crash could leave you with “meta says completed but DB write didn’t”.

Best practice:

Persist the instrumentation after the ingestion transaction commits and run status is updated to completed.

Store a run_id inside lastRunMeta and healthSummary, so you can correlate.

Ask:
