2) So why did ingest “abort”?

Because your advisory lock did its job… but your job coordination isn’t ideal yet.

Replit says:

“Another writer was holding the lock… ingest aborted.”

That means quick-score and ingest overlapped at the write moment.

Two important clarifications

The lock preventing the write is good — it prevents corruption.

But aborting means you might skip an hour’s publish, even though ingestion computed everything.

So the system is safe, but not perfectly scheduled.

3) How to stop this from happening again (best practice)

You want one of these patterns:

Option A (best): Ingest is the only publisher

quick-score never writes trending_people

quick-score can still compute/preview/debug, but only ingest publishes
This removes 95% of writer-lock collisions.

Option B: Same lock, but quick-score yields to ingest

If you want to keep both writers:

ingest should “win”

quick-score should skip its publish if ingest is near schedule

Example logic:

quick-score: if within ±10 minutes of ingest top-of-hour run → don’t write

or quick-score: try lock, if locked → silently exit (no harm)

Option C: Don’t abort ingest — wait briefly, then retry

Instead of aborting immediately:

try to acquire lock

if not available: wait 5–15 seconds with jitter, retry a few times

if still blocked: abort

This alone will make “missed publish” rare.
