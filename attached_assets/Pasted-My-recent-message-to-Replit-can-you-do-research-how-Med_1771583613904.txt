Yep — I did a quick dive into Mediastack’s docs and pricing, and there are a couple of important details that affect how you should structure calls for a 100-person leaderboard.

How Mediastack “keywords” really works (this matters a lot)

Mediastack’s keywords parameter is not a full boolean query language. It supports:

A phrase / sentence (spaces are allowed): keywords=food industry

Multiple comma-separated keywords (and it describes this as searching for keywords, plus allowing exclusions with -): keywords=food industry, meat

Exclusions: keywords=virus,-corona

Standard filters: sources, categories, countries, languages, plus limit (max 100) and offset.

What it doesn’t document is support for OR / parentheses syntax like ("Elon Musk" OR Musk Tesla). So if you’re currently passing “OR” style strings into keywords, you should assume it might not behave the way you expect (could be treated as literal text, or just degrade matching).

What this implies for your leaderboard design
1) Prefer one strong phrase per person (default)

For most people:

keywords = "<Full Name>" (as a phrase)

Add languages=en (unless you explicitly want multilingual)

sort=published_desc

limit=3 or limit=5 (you only need headlines + a count proxy)

This keeps results clean and reduces false matches.

2) Use conditional widening when counts look suspicious (best practice)

Elon is the perfect example: many articles say “Musk” without “Elon”.

Because Mediastack doesn’t give you boolean OR, the safest widening strategy (without blowing budget) is:

On refresh ticks only (every 2h):

Query A: keywords="Elon Musk"

If result count is below a threshold (say < 3), run Query B as a fallback:

keywords="Musk Tesla" or keywords="Musk SpaceX" (space-separated phrase)

Use whichever produces better coverage, and record it in diagnostics.

That gives Elon better coverage without doing 2 calls for everyone.

3) Be careful with comma-separated keywords

Docs show comma-separated keywords as supported, but they don’t clearly state whether it’s “AND” vs “OR” semantics.
So I’d avoid relying on commas for “OR broadening” and instead use the conditional second query method above.

Your call budget sanity check

On Mediastack Professional you get 50,000 requests/month.
At 100 people:

100 requests every 2 hours = 1,200/day ≈ 36,000/month (safe)

If you add conditional second queries for only a few people per refresh tick, you’re still fine.

If you ever go to 30-minute refresh: 100 * 48/day = 4,800/day ≈ 144,000/month → you’d need the 250k tier (as you already planned).

Concrete setup I’d recommend (for Replit)

Default per-person Mediastack query:

keywords = person.newsQueryOverride ?? person.name

languages = "en"

sort = "published_desc"

limit = 5 (or 3)

Conditional widening (only if low coverage on refresh tick):

If count < 3 and person is in “high-importance list” (Top 25 or canaries), do one more call using a widenedNewsQueryOverride you define per person.

Store in diagnostics:

provider, keywordsUsed, count, sample titles (you already do some of this)
