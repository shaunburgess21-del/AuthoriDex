Overall Assessment of Your AuthoriDex Trend Score Engine
Your engine is fundamentally well-architected. The back-and-forth with ChatGPT and Replit over the past 48 hours resolved several real systemic risks. Here's a layered analysis:

API-level null override: Returning null for change24h/change7d when baselineStatus !== "normal" is a solid safety net, even if stale values somehow persist in the DB.

LiveTick scheduler tight-loop: The negative-delay setTimeout was causing CPU spin and log flooding. Fixing it to always advance to the next future boundary is a must-have stability fix.

My Suggested Improvements to the Trend Score Engine
1. News Source Resilience (Most Urgent)
The biggest real-world risk exposed in your conversation is GDELT's brittleness. When news freshness drops to 0/100 people, the velocity signal collapses and the entire leaderboard goes red in unison — this is an engine reliability problem, not just an ops problem. The fix has two parts:
​

Add a fallback news provider: Use Serper News as a backup since you already have the Serper integration. If GDELT freshness drops below a threshold (e.g., <30% fresh people per run), automatically switch to the fallback for that run.

Add a "degradation governor": Cap how much a single source's freshness drop can move scores in one run. For example, if news goes from 78 → 0 fresh in one cycle, taper the news velocity contribution over 2–3 runs rather than treating it as an instant 100% drop. This prevents "0-news run = everyone red."

2. newsCoveragePct in baselineMeta
You currently surface baselineStatus, scoreVersion, baselineRunId, etc. in API responses. Add newsCoveragePct (and ideally wikiCoveragePct, searchCoveragePct) to that meta object. This makes it trivially easy to see in the network tab whether a "global red" event is source-driven — without any UI changes needed.
​

3. Per-Source Staleness Degradation Flags
Right now, when GDELT fails, the engine silently uses stale/zero news data and propagates the effect through scoring. You should add a sourceHealth object to each ingestion run and snapshot, such as:

json
{
  "wiki": { "fresh": true, "coveragePct": 100 },
  "news": { "fresh": false, "coveragePct": 0, "reason": "circuit_breaker" },
  "search": { "fresh": true, "coveragePct": 100 }
}
This transforms GDELT failures from invisible to observable, and gives your engine a first-class way to make decisions like "if news is degraded, reduce its weight contribution this run."

4. Hydration Coverage Hard Threshold
Replit added a coverage warning when hydratedCount < trackedCount, but a warning alone is too weak. If hydration fills fewer than 90% of expected people (e.g., 89/100), the engine should either:
​

Mark baselineStatus as degraded for that run, or

Fail the hydration and log it as a critical alert

Silently serving a 70% complete leaderboard looks "valid" in the UI but produces misleading rankings.

5. Velocity Taper Asymmetry Review
Your EMA uses asymmetric smoothing (slower on downward moves, alpha = 0.216). That's smart for preventing panic drops. However, if news goes completely stale, the taper may still allow scores to drop materially within one or two ingestion cycles. Consider adding a source-loss floor: if the news source was healthy in the previous run but drops to 0 fresh in the current run (i.e., a sudden blackout, not organic decay), clamp the velocity impact to a maximum single-run drop (e.g., -5% of velocity contribution max per run from news).
​

6. Admin Health Endpoint
You're currently debugging via network tab and Replit shell queries. A single /api/admin/health endpoint surfacing the following would save enormous time:

Field	Value
currentRunId + phase	wiki/gdelt/serper/write
baselineRunId + age	hours since baseline

trendingPeopleRowCount	100 expected

This is backend-only, no UI needed, but it prevents the situation where every "why is everything red?" requires a Replit shell investigation.

What Claude said that’s exactly right
1) Fallback news provider is the single biggest win

Yes. Your engine is only as stable as its least reliable upstream when that upstream has meaningful weight. A fallback (Serper News is perfect since you already pay and integrate Serper) turns “0/100 fresh news” from a leaderboard-wide event into a recoverable wobble.

2) Degradation governor

Also yes. Even with a fallback, you’ll eventually hit “sudden blackout” moments (timeouts, rate limiting, provider hiccups). A governor prevents “everything red” from a provider incident. That’s an “engine credibility” feature.

3) Coverage metrics in meta

100% yes. Putting newsCoveragePct (and wiki/search) into meta means you can diagnose in 10 seconds via network tab with no UI clutter.

4) Make staleness observable

Yes. If you don’t log “reason: circuit_breaker / rate_limit / timeout” you’ll keep wasting time debating whether drops are organic or system-driven.

5) Admin health endpoint

Yes, and it should include “phase” (wiki/news/search/write), last heartbeat age, and row counts.

So far: agree.

Where I’d tweak Claude’s recommendations
A) “Hydration hard threshold should fail or mark degraded”

I agree with the principle (don’t silently serve partial data), but I’d avoid “fail hydration” if it leaves you with nothing.

Better approach:

If hydration coverage < 90%:

Still hydrate, but set baselineStatus="degraded" and maybe dataCoveragePct=xx.

API already has the “null deltas when degraded” safety net — keep it consistent.

Optionally block movers widgets if coverage is too low (backend can return empty + reason), but don’t crash the whole system.

This keeps the site alive while being honest about quality.

B) “Velocity taper asymmetry / clamp sudden source-loss to max drop”

Good idea, but don’t over-engineer it. The cleanest version is:

Detect “provider incident”: newsCoverage drops sharply run-over-run (e.g., >50 percentage points) and previous run had normal coverage.

Apply a ramp-down to the news weight (not the whole velocity), over 2–3 runs.

This is simpler than clamping score deltas directly and avoids creating weird edge-cases where true organic declines get suppressed.

C) “Switch to paid if coverage <70% in >1/5 runs”

That heuristic is reasonable, but I’d track two metrics:

CoveragePct (how many people got any news signal)

Latency / cost per run (how long ingestion spends on news)

You might find GDELT is “acceptable coverage but expensive time-wise,” in which case Serper News primary might actually be better than a separate paid provider.
