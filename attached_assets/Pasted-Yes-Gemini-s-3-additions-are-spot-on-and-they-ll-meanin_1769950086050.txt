We are moving from placeholder cards to real production content for the Vote and Predict sections. We need a safe, repeatable content pipeline that supports:

Batch upload via CSV (initial seeding of many cards)

Admin panel to create/edit/manage cards one-by-one

Visibility controls (draft/live/archived and optional scheduling)

Hybrid image workflow (CSV creates draft rows; images attached later via Admin)

Aggregate Seed consistency (CRITICAL — see section 6)

We want a plan that integrates cleanly with the architecture we already built (including the seed aggregate pattern used for celebrity voting).

Vote

category (industry label)

timeline (No Deadline / 1 Week / 1 Month / Custom)

Predict

title

description (max 200)

MUST include: close_date / resolution_date rules

Native predictions (Weekly Up/Down, Top Gainer, etc.) are not in scope for batch upload in this prompt.

3) Goal / Objective

Design the system so we can:

Import many cards quickly (CSV → creates records safely)

Manage them safely (Admin panel CRUD + status controls + image attachments)

Keep database clean (no fake users)

Ensure markets can be settled (Admin resolves Real-World Markets)

4) Visibility / Status Requirements (Non-Negotiable)

Every content item (face-off, poll, market) must support:

status: draft | live | archived (or draft/scheduled/live/archived if you want scheduling)

Draft items never appear on the public frontend

Admin can toggle status without data loss

5) Hybrid Image Workflow (Non-Negotiable)

CSV cannot contain image files. We need a deliberate workflow:

Step 1: CSV import creates rows in DRAFT status (even if no images are present).
Step 2: Admin panel has a filter: “Missing images” (or “needs assets”).
Step 3: Admin uploads images (stored in Supabase Storage) and links them to the record.
Step 4: Admin flips status to LIVE when ready.

Image selection rules at render time:

If the entity is a leaderboard celebrity, use existing celebrity profile image.

If it’s a custom entity/topic, use the uploaded Supabase Storage image for that card.

You must propose:

Supabase bucket structure (e.g. vote-faceoffs/, vote-polls/, predict-markets/)

Stored DB fields (e.g. image_path, image_url, image_updated_at)

6) CRITICAL: “Aggregate Seed” Consistency (Non-Negotiable)

We already have a robust aggregate seed architecture elsewhere in FameDex.

You must plan these new voting/prediction content tables using the same pattern:

Use seed_count columns per option/outcome (e.g. seed_votes_a, seed_votes_b, seed_support_count, seed_neutral_count, seed_oppose_count, etc.)

Maintain separate real user vote tables (or reuse existing vote tables/endpoints)

UI totals = seed_* + real aggregates

Do NOT create fake user rows for seeding

Do NOT generate “seed users” or “seed vote records”

Your plan must explicitly show:

Which tables hold seed_* counts

Which tables hold real user votes

How totals are computed efficiently

7) CRITICAL: Real-World Markets MUST Have Resolution Mechanism

Unlike other cards, Real-World Markets end and must be settled.

Your plan MUST include:

Market statuses: open | closed | settled (or similar)

winning_outcome field (e.g. yes/no) once settled

settled_at, settled_by_admin_id fields

Admin settlement UI: “Settle Market” → choose winner → system updates market + frontend shows winning outcome & disables further predictions

No code now — just plan and schema.
