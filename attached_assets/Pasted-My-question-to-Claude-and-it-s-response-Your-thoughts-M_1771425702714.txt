Claude: 
It's a mix of both — there are genuine external constraints you can't control, but there's also a meaningful amount Replit can do on your side to make GDELT far more resilient. Here's the full picture:

What You Can't Control (Genuine External Factors)
GDELT's free DOC 2.0 API is intentionally rate-limited to protect its ElasticSearch clusters, and during high-traffic global news events (elections, disasters, major celebrity moments) rate limits tighten unpredictably. There is no SLA, no guaranteed uptime, no paid tier for the free API, and no way to pre-purchase capacity. 429 errors and timeouts are a structural reality of the free endpoint.

What Replit Can Do to Help Significantly
1. Smarter Request Spacing with Adaptive Backoff
Your current 5.5s fixed spacing between GDELT requests is a static guess. Replace it with exponential backoff with jitter:

On a 429 → wait baseDelay * 2^attempt + random(0, 1000ms)

On a 503/timeout → shorter wait (network issue, not quota)

On success → gradually decrease spacing back to 5.5s

This means during a quiet GDELT period you run fast; during rate-limit pressure you back off automatically instead of hammering and accumulating failures.
​

2. Priority Tiering — Don't Fetch All 100 Equally
Not all 100 people need fresh GDELT data equally urgently. Split into two tiers:

Tier 1 (top 25 by current fame_index): Always fetch GDELT, even if slow

Tier 2 (bottom 75): Fetch GDELT only if run is on schedule; otherwise accept stale/Serper fallback for this run

This means even a badly degraded GDELT run still produces fresh data for the people users actually look at.

3. Per-Run Time Budget with Graceful Cutoff
You already have some time budget logic (160s budget that was exhausted), but currently when GDELT is slow, it processes only 3 of 25 candidates before cutting off. The fix is:

Set a hard wall time per run (e.g., 90s for GDELT across all people)

When wall time is hit, mark remaining people as "news_source: cached" and proceed with Serper fallback for those people only

This guarantees the run always completes within a predictable window

4. GDELT Response Caching with Smart TTL
GDELT data doesn't change by the minute. If you fetched a valid result for a person in the last 90 minutes, reuse it for the current run rather than re-querying. Your circuit breaker already does some of this, but it should be more aggressive:

Cache per (person_id, query_string) with a 90-minute TTL

On GDELT degradation, extend TTL to 3 hours automatically

Only re-fetch if cache is expired and GDELT is healthy

This directly reduces your total GDELT query volume per run, which is the root cause of rate limit pressure.

5. Run GDELT at Off-Peak Times
GDELT's load spikes during US and EU business hours when major news events occur. Since your ingestion runs hourly:
​

Consider staggering the GDELT fetch phase to run at :15 or :45 past the hour rather than on the hour (when many other services also poll news APIs)

A 15-minute offset costs you nothing in data freshness but puts you in a quieter request window

6. Use GDELT's TIMESPAN Parameter More Efficiently
GDELT's DOC 2.0 API now supports searching up to a full year of coverage. Rather than making separate 24h and 7d queries per person (doubling your request count), you can make a single query with timespan=7d and derive both 24h and 7d counts from the same response by filtering returned article timestamps client-side. This cuts your GDELT request count roughly in half.
​

The Longer-Term Answer: GDELT Cloud
If GDELT reliability becomes a recurring issue, 
GDELT Cloud
 offers a paid, authenticated REST API with rate limiting and usage tracking designed for production workloads. It's backed by optimized infrastructure and claims sub-second queries. It would eliminate the "are we getting throttled?" uncertainty entirely. Worth evaluating once you have paying users to justify the cost.

Where I agree strongly, where I’d tweak, and one thing I’d fact-check:

What’s absolutely worth doing on your side
1) Adaptive backoff + jitter (highest ROI)

Yes. Fixed 5.5s spacing is brittle. You want something like:

on 429: exponential backoff + jitter

on timeout/5xx: backoff too, but maybe a different curve

on success streak: slowly decrease delay (but never below a safe floor)

This alone will turn “fail 0/100 fresh” into “partial freshness most runs.”

2) Time budget + graceful cutoff (keep runs finishing)

Yes. Put a hard wall clock for the GDELT phase (e.g., 60–120s), then:

stop calling GDELT,

mark remaining people as stale,

immediately use Serper fallback for those people (or cached values),

finish the run and write trending_people.

This prevents the “stuck run blocks everything” cascade.

3) Priority tiering (top N fresh)

Also yes. Users mostly care about the top of the board. Make sure top 25 (or top 50) get your best effort.

4) Smarter caching with TTL

Yes, but with nuance:

Cache the parsed counts (24h/7d) + “last fetched at”

Use a TTL like 60–120 minutes.

During degraded state, extend TTL and stop re-fetching aggressively.

This reduces request volume (which is your biggest enemy).

Where I’d tweak Claude’s suggestions
5) “Run at :15 or :45” (meh, but harmless)

This might help slightly, but it’s not a reliable lever. The real win is backoff + caching + time budget.

6) “Single 7d query → derive 24h client-side”

This is potentially good if the response includes timestamps for returned articles and you can reliably filter by last 24h. But two big caveats:

A 7d query might return a capped list of results that’s not representative of last 24h vs 7d (ranking/limits can skew).

If the API caps results (e.g., maxrecords), you can’t accurately infer 24h volume from a truncated 7d sample.

So: it can halve requests, but you must validate that derived 24h correlates well with the dedicated 24h query. If not, it will introduce misleading deltas/spikes.

One thing I would not treat as fact without checking
“GDELT Cloud paid API”

I’m not 100% sure about a “GDELT Cloud” paid product with SLA-like behavior. GDELT has been mostly “open project / public endpoints,” and there are also ways to query via BigQuery (GDELT 2.1 events, GKG) and other mirrors. Before planning around a paid GDELT tier, I’d verify what actually exists right now.
