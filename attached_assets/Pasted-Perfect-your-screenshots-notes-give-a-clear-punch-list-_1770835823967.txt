Perfect — your screenshots + notes give a clear punch list. I’m going to translate everything into exact fixes + how they should behave, so you can paste this to Replit and they implement without “interpretation drift”.

A) Dedicated market page issues (/markets/:slug)
1) Outcomes show 50/50 instead of your seed split

Likely cause: the detail page is calculating percentages from bets only (which are 0), or it’s defaulting to equal split when totals are 0, instead of using seeded values.

Fix required (core logic):

Each outcome needs a “base liquidity” value (your seed).

Percent/probability displayed should be based on:

displayAmount = seedVolume + userVolume

So the displayed probability becomes:

pct = displayAmount / SUM(all outcomes displayAmount)

Acceptance test: If Yes seed=135 and No seed=49 and there are no user bets, the page shows ~73% / 27% (exact rounding can be 73/27 or 73.4/26.6).

2) Total Pool shows 0 but should show seeded pool

Fix required:

Pool should be:

TotalPool = SUM(seedVolume) + SUM(userStakes)

If you seed 135 + 49, total pool should show 184 (or whatever unit you use: credits).

3) Participants show 0 but should include seed participants

Right now “Participants” is probably counting distinct bettors only (0).

Two good options (pick one):

Option A (recommended): Show two values:

“Participants: 0” (real users)

“Seed participants: X” (from admin seedCount)
This avoids misleading users.

Option B: Show combined:

“Participants: seedCount + uniqueBettors”
But this implies seed are “real” participants, which may be slightly misleading.

Your ask implies B, but I’d still recommend A for honesty + clarity.

Acceptance test: if seedCount=10 total, show Seed participants: 10 (even if real users are 0).

4) Closing date is wrong (shows Feb 10 2026 instead of Jun 30 2026)

Likely cause: the UI is using the wrong field:

using bettingClosesAt instead of closeDate

or using createdAt

or timezone parsing bug (less likely given it’s months off)

Fix required:

Define and consistently use:

closeDate / resolutionDate (the date the market resolves)

bettingClosesAt (optional earlier cutoff)

The badge “Ended” should be based on:

if market status is resolved/closed, OR

if bettingClosesAt passed (then “Betting Closed”), OR

if closeDate passed (then “Ended” only if you want automatic close)

Acceptance test: Selecting Jun 30 2026 in admin must render Jun 30, 2026 on page and card, and not show “Ended”.

5) Place Your Prediction container issues

You have three separate bugs here:

5.1 Selected outcome defaults to Yes even if you clicked No on the card

Fix: When user clicks a card button to open the detail page, pass a query param:

/markets/:slug?pick=no

Then the detail page:

reads pick

sets selected outcome accordingly

5.2 “Change” doesn’t let you reselect Yes/No

Fix: “Change” should toggle the outcome selector UI back on.
Simplest: the selector (Yes/No buttons) should always be visible on the detail page above Stake Amount.

5.3 “Place Prediction” button disabled / unclickable

This is usually because:

stake is required and empty

user auth required (not logged in)

market ended/closed

validation failing silently

Fix required:

Show explicit disabled-state reasons:

“Enter stake amount”

“Sign in to place prediction”

“Market closed”

If market is open and stake is valid and user is logged in, button must enable.

Acceptance test: Logged in user enters stake → button enables → places prediction (or shows expected confirmation).

6) Opening the URL only works by clicking Yes/No

You’re right — the heading should open the market.

Best UX (mobile-safe):

Make Title + Avatar + “Make Prediction” / “View Details” clickable to /markets/:slug

Do NOT make the entire card background clickable (people scroll/brush cards on mobile)

Add a clear CTA:

Replace the bottom “Make Prediction” (on multi) with a consistent:

“View Market” (opens /markets/:slug)

Keep outcome buttons as prediction actions

This avoids accidental navigation.

7) Card heights inconsistent across types

This is a common one, and you don’t want to solve it with a hard fixed px that truncates badly.

Recommended approach (best-looking + scalable):

Standardize the card into sections with consistent heights:

Header block: fixed height (title + teaser clamp)

Body block: flexible but bounded

Footer block: fixed height (pool + CTA/buttons)

Use line clamping for teaser:

Title: clamp to 2 lines

Teaser: clamp to 2 lines (wrap, don’t cut mid-line)

Multi-option card:

Show top 3 options by default + “+N more”

Keep the list container a fixed max height so it doesn’t stretch the card

Acceptance test: In a row of mixed cards, the bottoms align and the layout looks tidy.

marketAvatarType: 'celebrity' | 'upload' | 'preset' (preset optional)

marketAvatarCelebrityId (nullable)
