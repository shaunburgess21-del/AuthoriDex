Two things I’d still sanity-check (small but important)
1) “Bad run” logic should only increment streak when the news signal is expected to be fresh

If you count a run as “bad” just because news_count < 2, you might falsely streak people who genuinely have low coverage (obscure names, misspelled query, etc.).
You already mitigate this with “top 25 OR wiki/search above p50”, which helps a lot. Still, best practice is:

streak increments only if the provider used was “gdelt” (or primary provider) and it’s a “fresh window” attempt, not a cached fill-forward artifact

or if diagnostics include a newsAttempted: true / newsApiHit: true style marker

If you don’t have that marker, your qualification gate probably keeps it safe enough — but this is the one subtle source of “phantom streaks.”

2) Cooldown should be keyed to the fallback provider and the person

You want cooldown per person per provider (e.g., Serper), otherwise later when you add a paid provider you might accidentally block the better provider because Serper was recently used.

(If they implemented cooldown via “who had per-person fallback in last 90 minutes” without distinguishing provider, it’s still okay for now, but worth evolving.)

Is it worth implementing per-person fallback before paying for news?

Yes. Even if you adopt a paid news API, you’ll still have:

occasional entity coverage gaps,

transient provider hiccups,

query edge cases.

Per-person fallback becomes your reliability layer.
