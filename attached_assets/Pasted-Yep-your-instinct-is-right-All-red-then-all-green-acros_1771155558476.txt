Yep — your instinct is right. “All red, then all green” across everyone is almost never real-world signal. It’s almost always baseline selection + snapshot contamination + gaps.

Replit’s diagnostic is mostly on the money, but I’d tighten it into a clean, actionable “here’s what’s definitely wrong vs. what’s plausible”:

What I think is definitely wrong (high confidence)
1) Your 24h baseline is not anchored to a stable reference

If your “closest-to-24h-ago” logic can pick different snapshots depending on the time of day (especially across gaps), you’ll get the exact phenomenon you saw:

In the morning, “24h ago” lands near a high point → everything looks down.

Later, “24h ago” lands near a low point → everything looks up.

This alone can create system-wide sign flips without anything changing “in the world”.

✅ Fix is simple: baseline must be deterministic.

Best option: compute 24h change as ingestion-run vs ingestion-run, not “nearest timestamp”.

“Current run” vs “baseline run closest to 24h ago (within a strict window)”.

You’ve already got ingestion_runs and now run_id on snapshots — use that as the source of truth.

2) Live-tick must NOT write anything that affects canonical history

Replit’s claim that off-hour snapshots might be poisoning EMA/baselines is very plausible. Even if it happens rarely, it’s catastrophic, because the next ingestion run will pick up “latest fame_index” from history.

✅ Rule: only the hourly ingestion writes canonical snapshots.

Live tick should write only to:

trending_people.fame_index_live

trending_people.live_rank

trending_people.live_updated_at

optionally counters (views, etc.)

It should not write trend_snapshots at all. If you want a “live history”, make a separate table like live_ticks with a completely different pipeline.

3) Your baseline lookup must filter to “trusted snapshots”

Even if you keep old data around, your baseline comparisons must use only snapshots that are:

run_id IS NOT NULL

and belong to a completed ingestion run

This avoids:

off-hour tick writes

partial runs

weird “seconds apart” per-person timestamps

any other future pipeline that writes snapshots

✅ Make this the default everywhere (leaderboard, hot movers, detail page).

What might be true, but I’d verify
“EMA sawtooth pattern”

EMA itself doesn’t inherently produce a daily sawtooth unless something is resetting the state or the input is discontinuous (gaps or overwrites). So the sawtooth is likely a symptom of:

ingestion gaps

“latest fame_index” being grabbed from a polluted snapshot

or mixing different “flavors” of scores into the same historical stream

So I wouldn’t treat “EMA alpha too low” as the first fix. Get the data lineage clean first.

One note about “Hot Movers changing often”

Even with hourly ingests, Hot Movers can still change if:

you compute thresholds on every request (percentiles)

your underlying change24h values are unstable (baseline problem)

Once you anchor 24h to ingestion runs and cache the Hot Movers endpoint to the current run, it should stop flickering.
