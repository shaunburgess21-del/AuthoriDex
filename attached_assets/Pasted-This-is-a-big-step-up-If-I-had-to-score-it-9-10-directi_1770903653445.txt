The only parts I’d still sanity-check
A) “Pending” vs “Loading” semantics

Make sure “Pending” is a real state, not just “we got nothing back.”

A good rule:

Loading: request in-flight

Pending: backend says “not ready yet” (e.g., status=queued|generating)

Error: failure

Empty: truly not eligible / no data

If “Pending” is triggered just by missing data, it can become a confusing catch-all.

B) Cooling criteria clarity

Replit says: “score or rank dropping significantly.”
That’s fine, but ensure it doesn’t feel arbitrary:

If Cooling is rank-drop only but 24h % isn’t negative, users might go “why cooling?”

If Cooling is % drop only, it’s intuitive.

If you want it to be bulletproof, make Cooling require negative 24h change (and optionally rank drop), not rank drop alone.

C) Surging driver cue decision

Replit’s logic (“the tooltip already says Score spike vs Rank jump”) is mostly fine.

But your users will often not hover. On mobile, many won’t open tooltips either.

So the best compromise is tiny and non-invasive:

Keep the single “Surging” badge

Add a very small sublabel in Hot Movers rows only:

Surging · Score or Surging · Rank
This avoids cluttering the leaderboard while making the Hot Movers widget instantly understandable.

Not mandatory, but it’s high leverage.
