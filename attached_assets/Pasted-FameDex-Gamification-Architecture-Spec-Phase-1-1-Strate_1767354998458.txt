FameDex: Gamification & Architecture Spec (Phase 1)1. Strategic OverviewThis document outlines the architecture for the FameDex Phase 1 Gamification System. The core goal is to build a 'Service-Oriented' foundation that supports Virtual Credits today but allows for Real Money and Trust Scores in Phase 2 without refactoring.2. Architectural StandardsTo ensure scalability, the following patterns must be strictly enforced:Decoupled Permissions: UI components must never check raw XP. They must call checkPermission().Ledger System: All currency/XP changes must be recorded in transaction tables (credit_ledger, xp_ledger).Data-Driven Config: Game rules (XP amounts, Caps) must be stored in the DB, not hard-coded.3. Service Layer Logic (The "Black Box")Function: checkPermission(userId, permissionName)Fetch User's current Rank.Check ranks.capabilities JSONB column.Future Phase 2: Will also check users.trust_score and risk_level.Function: awardXP(userId, actionKey)Lookup xp_amount and daily_limit from xp_actions table.Count existing entries in xp_ledger for this user/action/today.IF count < limit: Insert Ledger Row & Increment User XP.IF count >= limit: Return success but do not award XP (Silent Cap).Rank Check: If new Total XP >= Next Rank Threshold, update user's rank_code.Function: getVoteWeight(userId, voteType)IF voteType == 'faceoff' OR 'poll': Return 1.0 (Strict Equality).ELSE (Induction/Curation): Return User's Rank Multiplier.
