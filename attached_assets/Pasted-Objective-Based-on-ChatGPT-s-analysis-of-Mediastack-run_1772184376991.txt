Objective
Based on ChatGPT's analysis of Mediastack run quality over the last 48h, implement 5 improvements to the ingestion health monitoring and reliability system:

Split Mediastack transport vs signal-quality health metrics (add top-10 nonZero, separate quality from transport)
Add a WEAK_COVERAGE state so fallbackOverride=true can no longer hide persistent Mediastack quality issues
Harden the duplicate hour-bucket guard (sequential duplicates are not blocked today, only concurrent ones)
Promote persistent selective fallback for known Mediastack zero-streak names (lower cooldown, proactive on refresh runs)
Add a 48h provider quality report API endpoint for admin diagnostics

T002: Add WEAK_COVERAGE state to Mediastack health tracking
Blocked By: [T001]
Details:
Create a new lightweight Mediastack-specific quality state machine persisted in api_cache under key system:mediastack_quality_state
Track: consecutiveWeakRefreshes (counter), state: "OK" | "WEAK_COVERAGE" | "RECOVERING"
Entry condition: top-25 nonZero% < 70% for 2 consecutive refresh runs (not cache-only runs)
Exit condition: top-25 nonZero% >= 70% for 1 refresh run
When in WEAK_COVERAGE: log a visible warning, include mediastackQuality: "WEAK_COVERAGE" in health summary
Do NOT remove fallbackOverride=true (it serves a real purpose for the OUTAGE state machine) — instead add this as a separate parallel quality track
Files: server/jobs/ingest.ts (add state persistence inline, similar to how NewsProviderPref is handled), server/scoring/sourceHealth.ts (optionally add helpers)
Acceptance: WEAK_COVERAGE warning fires when top-25 nonZero drops below 70% for 2+ consecutive refreshes. Clears on recovery.
