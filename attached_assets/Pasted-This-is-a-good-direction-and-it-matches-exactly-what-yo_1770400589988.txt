The one big concern: catch-up mode is in-memory

Replit said:

“server restarted… catch-up mode state is in-memory…”

That means restarts will keep re-triggering catch-up mode, which can be okay, but it can also cause oscillation if your server restarts often.

Fix (high ROI)

Persist catch-up mode state in DB (or even in your existing api_cache table):

catchUpActive

enteredAtHour

exitStreak count (consecutive runs below threshold)

So a restart doesn’t reset the streak logic and doesn’t cause flip-flopping.

Top 10 churn 0–2 is normal most hours
