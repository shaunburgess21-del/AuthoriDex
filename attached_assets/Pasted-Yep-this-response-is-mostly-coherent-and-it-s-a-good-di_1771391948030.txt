A stale lock / locked-out runs can explain stale data, but does not explain trending_people = 0 rows unless one of these is true:

It was genuinely first boot / new DB / table got wiped (or reset)

There is some separate cleanup job / migration / manual SQL that cleared it

The “stale rows are cleaned up at the end” logic can, in some edge path, delete too much

So the “snapshot-based fallback” is a great idea regardless — but you should still ask them to show the evidence of why it was empty.

✅ Is the snapshot-fallback plan beneficial?

Yes — absolutely. It’s a classic “never show an empty leaderboard” reliability pattern.

Even if the DB is perfect, a fallback prevents:

first deploy / first boot emptiness

edge-case empty state bugs

partial data loads

future regressions
